BoringStack is three repos that compose into the runtime, plus an optional fourth for OpenTofu-driven provisioning:

<FileTree>
- boringstack/
  - api-template/ Bun + Elysia + Drizzle API
  - ui-template/ Vite + React SPA
  - infra-docker-compose-template/ docker-compose stack
  - infra-bootstrap-tofu-template/ (optional) OpenTofu bootstrap
</FileTree>

Each repo has its own `package.json`, its own CI, its own deploy story (fork one and use it standalone if you want), and its own ESLint config tuned to the layer.

The three runtime repos wire together through the infra repo's `docker-compose.yml`, which references the API and UI via relative paths (`../../api-template`, `../../ui-template`).

## The disk layout

```
boringstack/                 ← parent dir; can have any name
├── api-template/
├── ui-template/
└── infra-docker-compose-template/
    └── compose/
        ├── docker-compose.yml
        │   #   build.context: ../../api-template
        │   #   build.context: ../../ui-template
        └── ...
```

<Aside type="caution" title="Renaming a fork">
  If you rename your fork of `api-template` or `ui-template`, either rename
  the cloned directory back to the canonical name or edit the
  `build.context` paths in
  `infra-docker-compose-template/compose/docker-compose.yml` once. The infra
  repo doesn't auto-discover; the paths are explicit.
</Aside>

## The contract between API and UI

The API exposes its OpenAPI document at `/swagger/json`. The UI ships a `pnpm generate:api` script that reads that document and rewrites `src/lib/api/schema.d.ts`. The UI's typed `apiClient` (built on `openapi-fetch`) refuses, at compile time, to call a path that isn't in the schema or pass a body that doesn't match.

So when the API gains, renames, or removes a route, the UI either fails to typecheck or generates the new contract on the next `pnpm generate:api`. There is no "I thought the field was `userName` not `username`" class of bug. TypeScript catches it before runtime.

See [OpenAPI client](/ui/openapi-client/) for the full flow.

## The contract between infra and the app repos

The infra `docker-compose.yml` defines `api-dev` / `api` / `ui-dev` / `ui` services whose build context is the sibling app repo and whose env vars are documented in `compose/api.dev.env.example` and `compose/api.prod.env.example`.

The api-template's `.env.example` and the infra's `api.*.env.example` overlap deliberately: same variable names, same shapes. If you move config between local-dev and the infra, the names don't change.

In prod the `build:` reference is a fallback. The default `image:` reference pulls a pre-built image from `ghcr.io/${IMAGE_OWNER}/${API_IMAGE_NAME}:${API_IMAGE_TAG}`, published by the api/ui repos' release workflows. See [Deployment](/topics/deployment/).

## Why not a monorepo?

[See "Why not a monorepo"](/architecture/why-not-monorepo/) for the long version. Short version: independent deploy lifecycles, separate dependency surfaces, and you should be able to fork just the API for a project that doesn't need a SPA.
