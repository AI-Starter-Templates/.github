This is the load-bearing piece of [Why BoringStack](/architecture/why-boringstack/): architecture lives in tooling, with prose for context.

Every template ships `AGENTS.md`, `CLAUDE.md`, and `AGENT_CONTRACT.md`. They explain the patterns: where routes go, when to use a service, how to write a Drizzle query, why `process.env` is forbidden outside the env validator.

Custom ESLint plugins make those patterns stick. Every rule that matters is machine-checked. `pnpm validate` / `bun run validate` is the merge gate: if lint fails, the diff does not ship. Prose is the why; lint is what ships.

<Aside type="tip" title="The hierarchy">
  Prose docs (`AGENTS.md`) are *aspirational*. Lint rules are *load-bearing*.
  When the two disagree, fix the lint rule or add a new one, then update
  the prose.
</Aside>

## How they're wired

Each repo's `eslint.config.ts` imports these plugins from `node_modules/` (they're installed as devDependencies, not vendored). The plugins ship their own `recommended` configs; we extend those and turn most rules to `error` rather than `warn` so the merge gate has real teeth.

```bash
# In each repo
pnpm validate     # typecheck + lint + knip + tests (merge bar)
pnpm lint         # lint only
pnpm lint:fix     # autofix what's autofixable
pnpm knip         # dead-code detection
```

Husky runs `pnpm lint-staged` on every commit, so most violations get caught before they reach CI. CI re-runs the full `validate` to catch what the staged-only run missed.

[Knip](https://knip.dev) sits alongside the lint plugins, surfacing unused files, unused exports, and unused dependencies. It runs as part of `validate` so dead code can't drift in unnoticed.

## Why this matters for agent-driven development

Lint turns architecture into a closed loop: code is written, violations surface as parseable errors, the fix is specific (wrong layer, missing schema, env read outside the validator). Review time stays on product logic instead of rediscovering folder rules.

For humans and agents alike, docs explain and lint enforces. Both matter; lint is what keeps the merge gate honest.

## Contributing back

The plugins are MIT-licensed and developed in the open. If you find a gap, file an issue or PR against the relevant plugin repo (linked above). Rules that prove themselves in production end up promoted to `recommended` configs.
