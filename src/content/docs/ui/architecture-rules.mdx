The ui-template enforces its architecture through ESLint, not vibes. The UI-specific work is [`eslint-plugin-react-component-architecture`](https://github.com/agjs/eslint-plugin-react-component-architecture); the rules below are what it catches, grouped by intent. It composes with the shared plugins (resource-architecture, module-boundaries, structured-logging, env-access, test-conventions, code-flow) that both repos use, plus ui-only plugins for TanStack Query cache shape and static i18n keys. Full inventory at [Lint as the contract](/architecture/lint-as-contract/).

## What it enforces

<FaqGroup>
  <FaqItem title="Component anatomy" open>
    No hooks in `.tsx`; logic belongs in `.hooks.ts`.
  </FaqItem>
  <FaqItem title="Single semantic module">
    Constants, utils, and types each get their own file (`.constants.ts`, `.utils.ts`, `.types.ts`).
  </FaqItem>
  <FaqItem title="className discipline">
    Use `cn(...)`; pull long class strings out for readable conditionals.
  </FaqItem>
  <FaqItem title="File naming">
    PascalCase components; kebab-case otherwise; suffix matches role (`.hooks.ts`, `.queries.ts`, `.store.ts`).
  </FaqItem>
  <FaqItem title="Prop ordering">
    Required props, optional props, then handlers.
  </FaqItem>
  <FaqItem title="View object boundary">
    Components read the hook's view object; no direct TanStack Query, Zustand, or `import.meta.env`.
  </FaqItem>
  <FaqItem title="Queries layer">
    `useQuery` only in `*.queries.ts`; component hooks call those query hooks.
  </FaqItem>
</FaqGroup>

Each rule has a fix-it suggestion where mechanical; the rest fail the lint gate and need a real edit.

## Component anatomy benefits

The anatomy makes any component mechanically testable. A `.tsx` with no state is pure JSX: a snapshot test or a single render assertion. A `.hooks.ts` returning a typed view object is testable via `renderHook`, no DOM needed. Together they give you full coverage of behavior without ever simulating a click.

The rules above are what keeps that split from collapsing back into one-big-`.tsx`-file under deadline pressure.

## Suppressing a rule

`// eslint-disable-next-line <rule>` works, but every suppression is reviewed. Common valid cases:

- Third-party render-prop APIs that force a hook-like pattern in `.tsx`.
- Sub-components that have no state but are too small to split into their own folder. Inline them; the lint rule has a size threshold.

If you find yourself suppressing the same rule across many files, that's a signal to update the rule, not to keep suppressing.
