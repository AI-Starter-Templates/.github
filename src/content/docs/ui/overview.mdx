The ui-template is a production-shaped SPA. The point of this template is not "look, a login page" — it's the **architecture rules around the login page** that make a team scale past three features without devolving into spaghetti.

<FileTree>

## The component anatomy

A page like `DashboardPage/` is **not** a single `.tsx` file. It's a folder with one job per file, all enforced by ESLint:

<FileTree>
- src/features/dashboard/components/DashboardPage/
  - DashboardPage.tsx              Pure JSX, no state, no queries
  - DashboardPage.hooks.ts         All `useState`, `useEffect`, `useCallback`
  - DashboardPage.types.ts         `IDashboardPageView` (the hook's return shape)
  - DashboardPage.constants.ts     Literals only — no imports
  - DashboardPage.utils.ts         Pure helpers
  - DashboardPage.test.tsx         Component test
  - DashboardPage.stories.tsx      Storybook
  - StatsSection.tsx               Sub-component (if needed)
  - StatsSection.types.ts          Its props
  - index.ts                       Re-export
</FileTree>

The lint config rejects `useState` inside a `.tsx`, rejects component files that import `*.queries.ts` directly, rejects multi-kind modules, etc. See [Architecture rules](/ui/architecture-rules/).

## State vs. server-state

| Kind | Lives in | Owner |
|---|---|---|
| **Server state** (anything fetched) | `*.queries.ts`, consumed via `useXxx()` hooks | TanStack Query |
| **UI / client state** (modal open, current step) | `*.store.ts` (Zustand) — feature-scoped | Zustand |
| **Form state** | `*.hooks.ts` with React Hook Form + Zod | RHF |
| **Render state** (purely computed) | `*.hooks.ts` returning `IXxxView` | none |

Components only see the **view object** from their hook. They never read TanStack Query directly, never read Zustand directly, never read `import.meta.env` directly.

## The typed OpenAPI client

The API exposes `/swagger/json`. The UI runs `pnpm generate:api` (against either the live api-dev or a saved spec) to regenerate `src/lib/api/schema.d.ts`. Then:

```ts
import { apiClient } from "@/lib/api/client";

const { data } = await apiClient.GET("/api/v1/users/me");
//      ^? typed exactly as the API's response shape
```

A call to a path that doesn't exist in the schema is a compile error. A body that doesn't match the OpenAPI shape is a compile error. The middleware in `src/lib/api/openapi.ts` handles 401 → silent refresh, then re-throws as `ApiError` for TanStack Query.

See [OpenAPI client](/ui/openapi-client/).

## Testing

| Layer | Tool | What it tests |
|---|---|---|
| Unit | Vitest + Testing Library | Hooks, utilities, schemas |
| Component | Vitest + Testing Library | One component + its hook |
| e2e | Playwright | Multi-page flows against the real stack |
| Visual | Playwright snapshots | Per-platform baselines |

The unit + component tests don't need a backend. Playwright e2e requires the dev stack running (`./dev.sh up -d` from infra). There's no MSW layer — we removed it deliberately; mocking a real backend is rarely worth the cost.
