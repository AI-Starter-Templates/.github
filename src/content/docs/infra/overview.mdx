The infra-docker-compose-template is the wiring that turns three repos into one running stack. Single-host VPS focus (Hetzner / OVH / DO / your laptop); a Kubernetes-flavored variant is planned as a separate repo.

## Service inventory

The base stack (always on) plus per-profile services:

| Service | Profile | Role |
|---|---|---|
| `postgres` (17-alpine) | always | App DB |
| `redis` (8-alpine) | always | Cache + queues |
| `traefik` (v3.7) | always | Reverse proxy + ACME |
| `api-migrate` | always (one-shot) | Schema push + demo user seed |
| `api-dev` | dev | Bun + Elysia API, bind-mounted source |
| `ui-dev` | dev | Vite dev server, bind-mounted source |
| `api` | prod | Built API image |
| `ui` | prod | Built UI image (nginx static serve) |

## Profile model

```bash
./dev.sh up -d                                 # baseline dev
STACK=prod ./dev.sh up -d                      # production: built images, HTTPS, ACME
WITH_OBSERVABILITY=1 ./dev.sh up -d            # + Prometheus/Grafana/Loki/Promtail/exporters
WITH_GLITCHTIP=1 ./dev.sh up -d                # + self-hosted error tracking
WITH_BULLMQ=1 ./dev.sh up -d                   # + queue dashboard at bullmq.localhost
WITH_WUD=1 ./dev.sh up -d                      # + image-update detection (Discord webhook)
```

Flags compose: `WITH_OBSERVABILITY=1 WITH_GLITCHTIP=1 ./dev.sh up -d` brings up both overlays alongside the base. See [Profiles & overlays](/infra/profiles-and-overlays/).

## Networks

Two bridge networks. Services are intentionally split:

| Network | Members |
|---|---|
| `backend` | postgres, redis, api[-dev], api-migrate, exporters, glitchtip, etc. |
| `frontend` | traefik, ui[-dev], api[-dev], glitchtip-web (HTTPS routing only) |

Traefik joins both, so it can route HTTPS from the `frontend` network into services that only live on `backend`. The data plane (Postgres, Redis) is never exposed to the `frontend` network.

## Resource budgets

Every service has `deploy.resources.limits` and `reservations` driven by env vars with sane defaults sized for a 4-vCPU / 8 GB host. Adjust in `compose/.env`; see [Resource limits](/infra/resource-limits/) for sizing guidance.

## What's NOT in this template

- **Kubernetes manifests.** Planned as a separate repo. This template is Compose-first; mixing Compose and cluster YAML in one repo is exactly the kind of confusion BoringStack tries to avoid.
- **Application code.** The api-template and ui-template own that.
- **Cloud-provider Terraform.** Out of scope; this is "what to run on a box," not "how to provision the box."
