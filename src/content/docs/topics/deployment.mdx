BoringStack ships a production path on a VPS you control: Docker Compose, GHCR images, Traefik TLS, Cloudflare at the edge. [Why BoringStack](/architecture/why-boringstack/) ties that to the rest of the templates.

Fork the templates, fill `compose/.env`, then `compose up -d` on the server, or use [OpenTofu](/topics/provisioning-with-tofu/) for first boot on Hetzner. HTTPS provisions via ACME. The perimeter uses Cloudflare plus a Hetzner cloud firewall.

## The mental model

Push to `main` on `api-template` or `ui-template`. The release workflow builds a Docker image and pushes it to `ghcr.io/<your-owner>/<repo>:latest`. WUD on the VPS notices the new tag and notifies you. You pull and restart with two commands. Traefik handles TLS via Let's Encrypt; Cloudflare proxies all traffic to your single apex domain via same-origin path routing.

## Design choices

<FaqGroup>
  <FaqItem title="Single-host first" open>
    One VPS carries real traffic for years; scale out when you need it.
  </FaqItem>
  <FaqItem title="Pre-built images from GHCR">
    Reproducible deploys; the VPS pulls images instead of running `npm install` on box.
  </FaqItem>
  <FaqItem title="WUD notify-only">
    You choose when to pull; no surprise restarts on a new tag.
  </FaqItem>
  <FaqItem title="Traefik + ACME">
    TLS on the origin without a separate load-balancer bill.
  </FaqItem>
  <FaqItem title="Same-origin path routing on the apex">
    One cert for `/` and `/api/*`; no `api.` subdomain.
  </FaqItem>
  <FaqItem title="Cloudflare-only ingress on 80/443">
    Only Cloudflare reaches the origin; port scans hit a closed firewall.
  </FaqItem>
  <FaqItem title="Multi-stage non-root `Dockerfile.prod`">
    Small runtime image without a shell in prod.
  </FaqItem>
</FaqGroup>

## First-time deploy

The preferred path is [Provisioning with OpenTofu](/topics/provisioning-with-tofu/): one `tofu apply` provisions the VPS, configures Cloudflare DNS + zone settings, and runs cloud-init which clones the infra repo, drops a rendered `compose/.env`, and pulls the GHCR images.

Manual path if you'd rather:

1. Provision a Ubuntu VPS with Docker installed.
2. Clone your `infra-docker-compose-template` fork onto the VPS.
3. Write `compose/.env` with `PUBLIC_UI_HOST`, `ACME_EMAIL`, `IMAGE_OWNER`, and the rest from `.env.example`.
4. `STACK=prod ./scripts/compose-up.sh pull && STACK=prod ./scripts/compose-up.sh up -d`.
5. Run the [Firewall & TLS runbook](/runbooks/firewall-and-tls/) to verify Cloudflare-only ingress.

You do **not** clone `api-template` or `ui-template` on the VPS. Their built images come from GHCR.

## Subsequent deploys

For changes to the api or the ui, you don't touch the VPS:

```bash
# On your laptop:
git push origin main
# release.yml builds + pushes ghcr.io/.../...:latest + :sha-<7> + :<semver>
```

WUD watches `latest` and posts a notification when a new tag appears. To apply:

```bash
ssh <vps>
cd /opt/boringstack/infra
docker compose pull
docker compose up -d
```

For changes to the infra YAML or env vars, `git pull` the infra repo on the VPS then re-run `compose up -d`.

## Image update strategy

<FaqGroup>
  <FaqItem title="WUD notify + manual pull (default)" open>
  `WITH_WUD=1`, Discord ping or dashboard at `:3033`, then pull and restart on the VPS when you are ready.
  </FaqItem>
  <FaqItem title="Hands-off auto-pull">
  Lower-stakes envs only. Swap to Watchtower or enable WUD's docker-compose trigger if you want unattended pulls.
  </FaqItem>
  <FaqItem title="Pin to a specific tag">
  Set `API_IMAGE_TAG=sha-abc1234` or `:0.3.0` in `compose/.env` and update deliberately.
  </FaqItem>
</FaqGroup>

The default ships notify-only because surprise restarts in production are a worse failure mode than a missed CVE notification. See [Image updates](/runbooks/image-updates/).

## Rollback

```bash
# On the VPS:
echo "API_IMAGE_TAG=sha-<previous-7>" >> compose/.env  # or a previous :<semver>
docker compose pull
docker compose up -d
```

Postgres schema is the only thing this doesn't roll back. Destructive migrations are forward-only by convention. The api-template's discipline is "additive changes are normal, destructive changes are deliberate," so most rollbacks just work.

For high-stakes deploys, snapshot Postgres before applying a destructive migration. See [Backups](/runbooks/backups/).

## Where to scale up

When single-host runs out of room, Postgres is almost always the bottleneck. In rough order:

1. Vertical: bigger VPS, bump `POSTGRES_LIMITS_*`.
2. Managed Postgres: Neon, Supabase, Crunchy, RDS. App stays on the VPS.
3. Replicate the API horizontally behind a real load balancer.
4. Kubernetes. Separate template planned; not in this repo because mixing Compose + cluster YAML is exactly the confusion BoringStack avoids.

## Provider notes

<FaqGroup>
  <FaqItem title="Hetzner Cloud" open>
    EU-friendly pricing and network. New accounts may sit in fraud review for a day or two.
  </FaqItem>
  <FaqItem title="OVH">
    Low cost with IPv4 included. Prefer the API over the web console for automation.
  </FaqItem>
  <FaqItem title="DigitalOcean">
    Polished UX and docs. Compare specs against Hetzner/OVH for your workload.
  </FaqItem>
  <FaqItem title="Your own metal">
    Lowest cost at scale; you operate the hardware and networking.
  </FaqItem>
</FaqGroup>

Cloudflare in front is standard regardless of provider; that's what makes the firewall + IP allowlist work.
