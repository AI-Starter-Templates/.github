BoringStack ships as four GitHub template repos. The flow is **"Use this template" → rename to taste → clone → boot**. The toolchain follows your rename: GHCR workflows, the prod compose stack, and the OpenTofu bootstrap derive image names and owners from your fork's actual URLs.

This page covers the few things that aren't fully automatic.

## Step 1. Fork the four repos

Click **"Use this template"** → **"Create a new repository"** on each:

- [`api-template`](https://github.com/AI-Starter-Templates/api-template) → e.g. `acme-api`
- [`ui-template`](https://github.com/AI-Starter-Templates/ui-template) → e.g. `acme-ui`
- [`infra-docker-compose-template`](https://github.com/AI-Starter-Templates/infra-docker-compose-template) → e.g. `acme-infra`
- [`infra-bootstrap-tofu-template`](https://github.com/AI-Starter-Templates/infra-bootstrap-tofu-template) → e.g. `acme-bootstrap`

Keep canonical names if you want. Nothing breaks either way.

<Aside type="note">
The docs site source (`.github` repo) is **not** a template. It's BoringStack-specific content. Skip it.
</Aside>

## Step 2. Clone them as siblings

Compose build contexts use relative paths (`../../api-template`, `../../ui-template`). Whatever you named your forks, lay them out as siblings:

```
acme/
├── acme-api/         (your fork of api-template)
├── acme-ui/          (your fork of ui-template)
├── acme-infra/       (your fork of infra-docker-compose-template)
└── acme-bootstrap/   (your fork of infra-bootstrap-tofu-template)
```

If you renamed api or ui, either rename the cloned directory to match what `compose/docker-compose.yml` expects, or edit the `build: context:` lines once.

## Step 3. Dev boot

```bash
cd acme-infra/compose
cp .env.example .env
./dev.sh up -d
open http://localhost:3001
```

Sign in as `demo@example.com` / `password123`.

## Step 4. Wire GHCR for prod

The `release.yml` workflows publish on push to `main` and on `v*.*.*` tag pushes. They use `${{ github.event.repository.name }}`, so they publish to `ghcr.io/<your-owner>/<your-renamed-repo>:latest` automatically. No edit needed.

The default `GITHUB_TOKEN` has `packages: write` via the workflow grant, so the first push from a freshly-forked repo works without secret setup.

**Make the GHCR package public** after the first publish:

1. Go to `https://github.com/<your-owner>/<your-renamed-repo>/pkgs/container/<your-renamed-repo>`
2. Package settings → Change visibility → Public.

Without this, downstream consumers (your VPS) need a pull credential.

## Step 5. Wire prod compose to your images

In `acme-infra/compose/.env`:

```bash
# Renamed repos:
IMAGE_OWNER=acme
API_IMAGE_NAME=acme-api
UI_IMAGE_NAME=acme-ui
```

Kept canonical names? Only `IMAGE_OWNER` needs to change.

## Step 6. Wire OpenTofu bootstrap

In `acme-bootstrap/terraform.tfvars`:

```hcl
api_repo   = "https://github.com/acme/acme-api"
ui_repo    = "https://github.com/acme/acme-ui"
infra_repo = "https://github.com/acme/acme-infra"
domain     = "acme.com"
# ... other vars per terraform.tfvars.example
```

The module derives `IMAGE_OWNER`, `API_IMAGE_NAME`, and `UI_IMAGE_NAME` from those URLs and renders them into `compose/.env` on the VPS. You don't set them twice.

## Checklist

- [ ] All four template repos forked under your org/user
- [ ] First image published to GHCR (push to `main`, watch the Actions tab)
- [ ] GHCR package visibility set to Public
- [ ] `acme-infra/compose/.env`: `IMAGE_OWNER` + `*_IMAGE_NAME` (if renamed)
- [ ] `acme-bootstrap/terraform.tfvars`: fork URLs + real `domain`
- [ ] `demo@example.com` deleted or rotated before any production traffic
- [ ] `JWT_SECRET` regenerated for production
- [ ] `ACME_EMAIL` set to a real address (not `example.com`)
