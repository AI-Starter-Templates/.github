## TL;DR

1. Click **"Use this template"** on the four BoringStack repos under your org.
2. Clone your four forks as siblings under one parent directory.
3. `cd <infra-fork>/compose && cp .env.example .env && ./dev.sh up -d`
4. Open `http://localhost:3001` and sign up.

You now have Postgres, Valkey, the API, and the UI running locally with hot reload. The toolchain follows your repo names: GHCR workflows, the prod compose stack, and the OpenTofu bootstrap derive image names and owners from your fork URLs automatically. You only configure your renames when you go to prod (see [Going to production](#going-to-production)).

## Prerequisites

- Docker + Docker Compose v2 (`docker compose version` reports `v2.x` or newer).
- About 4 GB of free RAM.
- No local Node or Bun needed; everything runs in containers.

## Step 1. Use this template

BoringStack ships as four GitHub template repos. Click **"Use this template" → "Create a new repository"** on each:

- [`api-template`](https://github.com/AI-Starter-Templates/api-template) → e.g. `acme-api`
- [`ui-template`](https://github.com/AI-Starter-Templates/ui-template) → e.g. `acme-ui`
- [`infra-docker-compose-template`](https://github.com/AI-Starter-Templates/infra-docker-compose-template) → e.g. `acme-infra`
- [`infra-bootstrap-tofu-template`](https://github.com/AI-Starter-Templates/infra-bootstrap-tofu-template) → e.g. `acme-bootstrap`

Keep the canonical names if you want. Nothing breaks either way.

<Aside type="note">
The docs site source (`.github` repo) is **not** a template. It's BoringStack-specific content. Skip it.
</Aside>

## Step 2. Clone your forks as siblings

The compose build contexts use relative paths (`../../api-template`, `../../ui-template`). Whatever you named your forks, lay them out as siblings under one parent:

```bash
mkdir acme && cd acme

git clone https://github.com/<you>/acme-api
git clone https://github.com/<you>/acme-ui
git clone https://github.com/<you>/acme-infra
git clone https://github.com/<you>/acme-bootstrap
```

```
acme/
├── acme-api/         (your fork of api-template)
├── acme-ui/          (your fork of ui-template)
├── acme-infra/       (your fork of infra-docker-compose-template)
└── acme-bootstrap/   (your fork of infra-bootstrap-tofu-template)
```

If you renamed `api-template` or `ui-template`, either match the cloned directory name to what `compose/docker-compose.yml` expects, or edit the `build: context:` lines once.

<Aside type="tip" title="Optional: setup.sh">
The workspace root ships a `setup.sh` that bootstraps `compose/.env`, generates a `GLITCHTIP_SECRET_KEY`, and (with `--up`) boots the stack. Skip if you prefer doing the steps by hand.
</Aside>

## Step 3. Boot the local stack

<Steps>

1. Seed the env:

   ```bash
   cd acme-infra/compose
   cp .env.example .env
   ```

2. Make scripts executable:

   ```bash
   chmod +x dev.sh ../scripts/*.sh
   ```

3. Run:

   ```bash
   ./dev.sh up -d --build
   ```

   First boot pulls base images, builds the api/ui dev images, runs migrations. If you set `SUPERUSER_EMAIL` + `SUPERUSER_PASSWORD` in `compose/.env`, an admin user is also created. About 3 minutes on a fast laptop.

</Steps>

## Step 4. Sign in

Open **http://localhost:3001**.

If you set `SUPERUSER_EMAIL` + `SUPERUSER_PASSWORD` in `compose/.env` before booting, sign in with those credentials. Otherwise hit "Sign up" on the form and register a new account; that user is the first user in the system. You land on the dashboard, which is intentionally minimal. Start building from here.

## What's running

<FaqGroup>
  <FaqItem title="postgres (port 5432)" open>
    App database; schemas `auth`, `billing`, `audit`, `app`, `notifications`.
  </FaqItem>
  <FaqItem title="valkey (port 6379)">
    Cache and BullMQ queues.
  </FaqItem>
  <FaqItem title="api-migrate (one-shot)">
    `db:push` and optional superuser seed, then exits.
  </FaqItem>
  <FaqItem title="api-dev (port 3000)">
    Bun + Elysia API with hot reload via bind mount.
  </FaqItem>
  <FaqItem title="ui-dev (port 3001)">
    Vite dev server; proxies `/api/*` to `api-dev`.
  </FaqItem>
</FaqGroup>

Traefik runs only in the prod profile. In prod it terminates TLS and path-routes `/api/*` + `/health` to the api container, everything else to ui, on one domain.

## Going to production

A second phase, not part of the 5-minute path. When you're ready to deploy your forks to a VPS:

1. Push to `main` on your api / ui forks. The release workflow builds and publishes `ghcr.io/<your-owner>/<your-repo>:latest` automatically; no secret setup needed for the default `GITHUB_TOKEN`.
2. After the first publish, make the GHCR package **public** (`github.com/<owner>/<repo>/pkgs/container/<repo>` → Settings → Change visibility) so the VPS can pull without credentials.
3. Set `IMAGE_OWNER` (and `*_IMAGE_NAME` if you renamed) in your infra fork's `compose/.env`. The OpenTofu bootstrap derives these from your fork URLs in `terraform.tfvars`, so they don't need to be set twice.

See [Deployment](/topics/deployment/) for the full prod path and [Provisioning with OpenTofu](/topics/provisioning-with-tofu/) for first-time VPS provisioning.

## First-deploy checklist

- [ ] Every template repo forked under your org/user
- [ ] First image published to GHCR (push to `main`, watch the Actions tab)
- [ ] GHCR package visibility set to Public
- [ ] `acme-infra/compose/.env`: `IMAGE_OWNER` + `*_IMAGE_NAME` (if renamed)
- [ ] `acme-bootstrap/terraform.tfvars`: fork URLs + real `domain`
- [ ] `SUPERUSER_PASSWORD` rotated via the password-reset flow after first login
- [ ] `JWT_SECRET` regenerated for production
- [ ] `FRONTEND_URL` and `PUBLIC_API_URL` match the production origin
- [ ] `ACME_EMAIL` set to a real address (not `example.com`)
